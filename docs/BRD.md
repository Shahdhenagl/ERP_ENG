## 1. الملخص التنفيذي

City Engineering شركة تبيع أجهزة UPS، تركّبها، وتتعاقد على صيانتها. النظام الحالي يوحّد ثلاث دورات كانت منفصلة (تجاريًا، مخزنيًا، وفنيًا) في نظام واحد يدور حول **سجل الجهاز بالرقم التسلسلي**.

النظام يتكوّن من:

1. **واجهة برمجية (API)** — Laravel 12 + Sanctum، هي المصدر الوحيد للحقيقة ولقواعد العمل.
2. **تطبيق المدير** — واجهة ويب كاملة تحت `/manager` تغطي كل الموديولات (تجاري، مخزني، مالي، محاسبي، خدمي).
3. **تطبيق الفني** — تطبيق موبايل مثبّت (PWA) تحت `/tech`، مساحة عمل مختصرة تخدم فنيًا واقفًا أمام الجهاز.

الثلاثة **قاعدة كود واحدة ونشر واحد** — لا يوجد تطبيقان منفصلان يجب مزامنتهما، بل منطقتا عناوين (URL areas) تشتركان في نفس المكوّنات وتفترقان في الصلاحيات والشاشات.

## 5. أصحاب المصلحة والأدوار

| الدور | القيمة في النظام | ما يملكه |
|---|---|---|
| **مدير النظام** `admin` | كل ما يملكه المدير + دليل الحسابات والقيود اليدوية وإدارة المستخدمين والإعدادات |
| **المدير / موظف المكتب** `manager` | إنشاء المهام والإسناد، العملاء، الأصول، العقود، المخزون، المشتريات، المبيعات، الفواتير، الخزنة، قراءة القوائم المالية |
| **الفني** `technician` | مهامه هو فقط: قبول، تحرّك، تنفيذ، تقرير فني، مرفقات، توقيع العميل، قراءة عهدته وتاريخ الجهاز الذي أمامه |
| **العميل** | لا يدخل النظام حاليًا — يستقبل واتساب/بريدًا ويوقّع على شاشة الفني |

**قاعدة الفصل:** الصلاحية تُطبَّق على مستوى **المسار** (middleware `role`) و**الصف** معًا — الفني الذي يفتح مهمة ليست له يُرفض في الـ Controller لا في الواجهة فقط.

## 6. المعمارية العامة

```
┌──────────────────────────────┐   ┌──────────────────────────────┐
│  تطبيق المدير  /manager      │   │  تطبيق الفني  /tech          │
│  React 19 · شاشة مكتب        │   │  React 19 · PWA مثبّت        │
└──────────────┬───────────────┘   └───────────────┬──────────────┘
               │            Bearer Token (Sanctum)  │
               └──────────────┬─────────────────────┘
                              ▼
              ┌───────────────────────────────┐
              │   Laravel 12 REST API         │
              │   Controllers → Services      │
              │   (كل قاعدة عمل في Service)   │
              └───────────────┬───────────────┘
                              ▼
                     MySQL 8 · Queue · Storage
```

**مبدأ معماري ملزم:** كل تغيير حالة أو أثر مالي يمرّ عبر **خدمة واحدة مالكة** — لا يكتب أي Controller حقل `status` أو يُنشئ قيدًا مباشرة.

| الخدمة | مسؤوليتها الحصرية |
|---|---|
| `TaskWorkflow` | انتقالات حالة المهمة + السجل + الإشعارات |
| `StockLedger` | كل حركة مخزنية وحساب المتوسط المرجح المتحرك |
| `CustodyService` | العهد الثلاثة (قطع / نقدية / أجهزة) |
| `BillingService` | إنشاء الفواتير وترحيلها |
| `SalesService` / `PurchasingService` | دورتا البيع والشراء |
| `Ledger` / `LedgerPoster` / `LedgerBackfill` | القيود المزدوجة وترحيل المستندات |
| `MaintenancePlanner` / `ContractSla` | توليد الزيارات وحساب الالتزام |
| `WarrantyService` | صلاحية الغطاء وقرار المطالبة |
| `FinancialReports` / `TreasuryReport` | القوائم والتقارير الفترية |

```

                                                    ▼
Item ──► StockLevel ──► Warehouse ──► Custody ──► Technician
                                                    │
Customer ─► Branch ─► Asset (سيريال) ◄──────────────┘

                       ├─► Contract ─► ContractVisit ─► Task
                       └─► Task ─► TaskReport + TaskAttachment + TaskStatusLog
                                      │
Quotation ─► SalesOrder ─► Invoice ◄──┘
                              │
                              ▼
                        Payment ─► CashMovement ─► CashBox
                              │
                              ▼
                     JournalEntry ─► JournalLine ─► Account / CostCenter
```

**الأصل (`Asset`) هو محور النظام** — كيان مستقل عن الصنف (`Item`)، له سيريال وحالة (`active` / `under_repair` / `retired`) وتاريخ كامل.

## 8. المتطلبات الوظيفية بالموديولات

### 8.1 الهوية والصلاحيات — `M-AUTH`

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| AUTH-1 | دخول ببريد وكلمة مرور، رمز Bearer | محاولة فاشلة متكررة تُخنق عند 10/دقيقة |
| AUTH-2 | ثلاثة أدوار محدَّدة في `UserRole` | مستخدم موقوف لا ينفذ أي طلب |
| AUTH-3 | كل مستخدم يغيّر بياناته وكلمة مروره | صفحة `Profile` لكل الأدوار |
| AUTH-4 | إدارة المستخدمين لمدير النظام فقط | `role:admin` على `users.*` |
| AUTH-5 | سجل تدقيق `activity_logs` لكل إنشاء/تعديل/تغيير حالة | كل عملية لها فاعل ووقت |

### 8.2 العملاء والفروع — `M-CRM`

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| CRM-1 | ملف عميل: بيانات، هاتف، واتساب، إحداثيات | زر «الاتجاهات» يفتح الخرائط |
| CRM-2 | العميل الواحد له عدة **فروع/مواقع** | الجهاز والمهمة يرتبطان بفرع بعينه |
| CRM-3 | **كشف حساب العميل** لفترة | `GET customers/{id}/statement` + نسخة قابلة للطباعة |
| CRM-4 | الفني يقرأ كارت العميل لمهمته فقط | القراءة مسموحة، التعديل ممنوع |

### 8.3 سجل الأصول — `M-ASSET`

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| AST-1 | الجهاز كيان مستقل بسيريال، لا حقول نصية على المهمة | ترحيل البيانات القديمة تم في `..._000008` |
| AST-2 | ربط الجهاز بالعميل والفرع والماركة والموديل والقدرة وتاريخ البيع | — |
| AST-3 | **سجل صيانة كامل** لكل جهاز على `/manager/assets/{id}` | كل مهمة وتقرير ومطالبة تظهر في مكان واحد |
| AST-4 | حالة الضمان لها **ثلاث قيم** لا اثنتان: ساري / منتهي / **غير محدد** | «غير محدد» لا يُعامَل انتهاءً في أي شاشة |
| AST-5 | الفني يقرأ تاريخ الجهاز الذي أُرسل إليه فقط | الـ Controller يرفض أصلًا لم يُندَب إليه |

### 8.4 أوامر العمل والتشغيل الميداني — `M-WO`

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| WO-1 | أمر عمل مرقّم `WO-2026-0001` بنوع وأولوية وموعد وجهاز | خمسة أنواع: تركيب/صيانة/إصلاح/فحص/تسليم |
| WO-2 | **آلة حالة محكومة**: بانتظار ← مقبولة ← في الطريق ← جارٍ العمل ← منتهية / ملغاة | لا تخطّي خطوة، ولا إعادة فتح منتهية |
| WO-3 | تقدّم المهمة **يملكه الفني وحده** | المدير يسند ويلغي، ولا يقود التنفيذ |
| WO-4 | كل انتقال يُسجَّل بالفاعل والوقت والإحداثيات | `task_status_logs` |
| WO-5 | تقرير فني بقراءات **مُهيكلة رقميًا** (جهد دخول/خروج، تحميل، بطاريات، زمن Backup) | لا نص حر — لتمكين تحليل الاتجاه لاحقًا |
| WO-6 | مرفقات صور قبل/بعد من كاميرا الموبايل | رفع وحذف |
| WO-7 | توقيع العميل على الشاشة | يُحفظ مع التقرير |
| WO-8 | قطع الغيار المستخدمة تُخصم من **عهدة الفني** لا من المخزن الرئيسي | ربط `TaskReport` بحركة مخزنية |
| WO-9 | إشعار عند الإسناد وعند كل تغيير حالة: داخل النظام + بريد + **Web Push** | يصل والتطبيق مغلق |
| WO-10 | روابط واتساب جاهزة: مدير→فني، فني→مدير، مدير→عميل | عبر `wa.me` بلا تكلفة رسائل |
| WO-11 | فشل الإشعار **لا يُلغي** العملية التي يعلن عنها | العمل يُحفظ ثم يُشعَر |

### 8.5 عقود الصيانة — `M-CONTRACT`

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| CTR-1 | عقد سنوي يغطي **عدة أجهزة** (`asset_contract`) | — |

### 8.6 الضمانات والمطالبات — `M-WARRANTY` *(منفَّذ · غير مُلتزَم بعد)*

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| WRT-1 | ثلاثة أنواع غطاء: **ضمان الشركة / ضمان المورّد / تمديد مدفوع** | `WarrantyKind` |
| WRT-2 | الغطاء **لا يُعدَّل بتغيير تواريخه** — يُمدَّد أو يُلغى بعملية موثّقة | `extend` / `void` فقط |
| WRT-3 | مطالبة بدورة: مفتوحة ← معتمدة/مرفوضة ← تم الإصلاح/الاستبدال ← مغلقة | `ClaimStatus` |
| WRT-4 | المطالبة المعتمدة تُولّد **أمر عمل إصلاح** | `claims/{id}/repair-order` |
| WRT-5 | «تاريخ الجهاز» يجمع الغطاء والمطالبات وأوامر الإصلاح في عرض واحد | `assets/{id}/history` |

| الرقم | المتطلب | معيار القبول |
|---|---|---|

| INV-2 | مخازن متعددة بأنواع (رئيسي / سيارة فني / …) | `WarehouseType` |

| INV-5 | شاشة المخزون **أقسام** لا تبويبات: أصناف / مخازن / عهد / حركات | الشريط الجانبي يفتح القسم مباشرة |

### 8.8 العهد — `M-CUSTODY`

| الرقم | المتطلب | معيار القبول |
|---|---|---|

| CUS-2 | عهدة النقدية: صرف من الخزنة ← إنفاق بمستند | `custody/cash` + `custody/spend` |

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| ACC-1 | دليل حسابات هرمي بأنواع | `AccountType` · `ChartOfAccounts` |
| ACC-2 | مراكز تكلفة | لقياس ربحية الخدمة |
| ACC-3 | **كل مستند يُرحَّل آليًا** إلى قيد مزدوج | `LedgerPoster` · `JournalSource` يحدد المصدر |
| ACC-4 | قيد يدوي وعكس قيد — **لمدير النظام فقط** | لا حذف صامت، عكس موثّق |
| ACC-5 | ترحيل استدراكي للمستندات السابقة على وجود الدفتر | `accounting/post` (`LedgerBackfill`) |
| ACC-6 | تقارير: ميزان مراجعة · قائمة دخل · مركز مالي · أستاذ عام | `FinancialReports` |
| ACC-7 | القوائم **تُقرأ** بصلاحية المدير، والدفتر **يُكتَب** بصلاحية مدير النظام | الفصل مطبَّق على مستوى المسار |

### 8.13 المستندات المطبوعة — `M-PRINT`

| الرقم | المتطلب | معيار القبول |
|---|---|---|
| PRN-1 | أربعة مستندات: فاتورة · عرض سعر · تقرير خدمة · كشف حساب | خارج إطار الشاشة — ورقة لا واجهة |
| PRN-2 | ترويسة الشركة من الإعدادات | يقرأها **كل** الأدوار (الفني يطبع في الموقع) |
| PRN-3 | تقرير الخدمة بمسار غير مقيّد بمنطقة | الفني والمدير يفتحان نفس الرابط |

### 9.1 قواعد التنقّل

ثلاث قواعد تحكم كل الشاشات، وكل انحراف عنها اعتُبر خطأً وأُصلح:

| القاعدة | السبب |
|---|---|
| الموديول متعدد الأقسام = **مسارات** لا تبويبات داخلية | الشريط الجانبي يفتح القسم المطلوب مباشرة، والرابط يُرسَل لزميل فيصل لنفس الشاشة |

| الأب في الشريط لا يتكرّر كابن | صفّان يشيران لمسار واحد يُضيئان معًا |

### 9.2 مسار العمل — تطبيق المدير

**أ. اليوم المعتاد**

```
لوحة المعلومات
  ├─ «متأخرة» ──────► المهام (مفلترة) ──► تفاصيل المهمة ──► إسناد / متابعة
  ├─ «غير مُسنَدة» ──► إسناد فني ────────► إشعار للفني (داخلي + بريد + Push)
  └─ «زيارات مستحقة» ► تحويل الزيارة إلى أمر عمل
```

**ب. من بلاغ إلى نقود** — المسار الذي يمرّ عبر أغلب الموديولات:

```
① مهمة جديدة ──► العميل ──► الفرع ──► الجهاز (سيريال) ──► النوع والأولوية والموعد ──► إسناد فني
                                        │
                                        └─► اختيار الجهاز يُظهر **حالة ضمانه** بجواره فورًا
                                            (ساري / منتهي / غير محدد) — النظام يعرض، والمدير يقرّر
                                            هل الزيارة محاسَبة. الربط الآلي بين الغطاء والفوترة
                                            لم يُنفَّذ بعد (راجع R-11).
② الفني ينفّذ (راجع 10.2) ──► المهمة تصبح «منتهية» ──► إشعار للمدير
③ تفاصيل المهمة ──► «فوترة المهمة» ──► فاتورة مسودة
      · قطع الغيار المستهلكة تنزل تلقائيًا بسعر التكلفة
      · سطر «أجر زيارة» يظهر بصفر — قرار السعر للمدير لا للنظام
④ تسعير ──► «إصدار» ──► الفاتورة تصبح مستندًا لا يُعدَّل (يُلغى فقط)
⑤ «تسجيل تحصيل» ──► اختيار الخزنة وطريقة الدفع ──► سند قبض
⑥ الخزينة ──► رصيد الخزنة تحرّك ──► كشف الخزنة يعرض الحركة
⑦ المحاسبة ──► القيود اليومية تحوي قيدين جديدين (الفاتورة + السند) بلا تدخّل
```

**ج. مسارات الموديولات**

| الموديول | المسار على الشاشة |
|---|---|
| **المبيعات** | عرض سعر (مسودة) ← إرسال ← قبول العميل ← **أمر بيع** ← تسليم (يُخرج المخزون) ← فوترة |
| **المشتريات** | مورّد ← أمر شراء (مسودة) ← إرسال ← استلام (كامل أو جزئي، يُدخل المخزون ويحرّك التكلفة) ← سداد |
| **المخزون** | صنف ← استلام في مخزن ← **تحويل إلى سيارة فني** (= تسليم عهدة) ← استهلاك على مهمة ← تسوية جرد |
| **العقود** | عقد ← ربط الأجهزة ← **تفعيل** (يولّد خطة الزيارات) ← الزيارة تصير أمر عمل ← قياس SLA |
| **الضمانات** | تسجيل غطاء على جهاز ← شهادة مطبوعة ← **مطالبة** ← فحص ← اعتماد/رفض ← أمر إصلاح ← إغلاق |
| **المحاسبة** | دليل الحسابات ← الأستاذ العام لحساب بعينه ← ميزان المراجعة ← قائمة الدخل ← المركز المالي |

**د. قواعد تفاعل ثابتة عبر كل الشاشات**

- **الفترة تُختار مرة واحدة** على مستوى الموديول لا الشاشة — الخزينة والمحاسبة يقرأ كل أقسامهما نفس النافذة الزمنية، فيستحيل أن يظهر ميزان مراجعة لشهر بجوار قائمة دخل لشهر آخر.
- **التاريخ المكتوب يتغلّب على الاختصار** («هذا الشهر» / «هذه السنة») — وإلا لمَسح الاختصار ما كتبه المستخدم للتو.
- **كل عملية مالية تُثبِت أثرها فورًا** في الشاشة نفسها: تسجيل مصروف يحدّث رصيد الخزنة والميزان معًا، لأن الاستعلامات تُبطَّل كمجموعة واحدة.
- **الخطأ يُعرَض على الحقل** لا في رسالة عامة (`fieldErrors`)، والنجاح يُعرَض كـ Toast ثم يُغلق الحوار.

## 10. تطبيق الفني — الشاشات

**المسار:** `/tech` · **المستخدم:** الفني · **الشكل:** PWA مثبّت على شاشة الموبايل الرئيسية، عربي RTL.

| الشاشة | المسار | المحتوى |
|---|---|---|
| لوحته | `/tech` | مهام اليوم وما ينتظر قبوله |
| مهامي | `/tech/tasks` | قائمة مهامه هو فقط |
| تفاصيل المهمة | `/tech/tasks/:id` | القبول · «في الطريق» · بدء العمل · الاتجاهات · واتساب المدير · التقرير الفني · الصور · توقيع العميل · الإنهاء |
| الجهاز | `/tech/assets/:id` | تاريخ الجهاز الذي أمامه (قراءة فقط) |
| عهدتي | `/tech/stock` | ما في سيارته من قطع (قراءة فقط) |
| ملفي | `/tech/profile` | تغيير كلمة المرور |
| تقرير الخدمة | `/print/tasks/:id` | يطبعه في الموقع ويوقّعه العميل |

**سلوكيات خاصة بالفني:**
- **إشعار Push** يصل والتطبيق مغلق، والضغط عليه يفتح المهمة مباشرة (Service Worker).
- **الإحداثيات** تُلتقط مع كل تغيير حالة — إثبات وصول.
- **الكاميرا** مصدر المرفقات مباشرة.
- الفني **لا يرى** الشريط الجانبي الإداري ولا أي شاشة مالية؛ فتحه `/manager` يعيده إلى `/tech` بصمت لا بصفحة خطأ.

### 10.1 آلة الحالة — الحدود المسموح بها

```
   بانتظار القبول
        │  قبول
        ▼
    تم القبول ─────────────┐
        │  في الطريق        │  بدء العمل مباشرة
        ▼                   │  (زيارة داخل الورشة)
    في الطريق ──────────────┤
        │  بدء العمل        │
        ▼                   ▼
        جارٍ العمل ◄────────┘
        │  إنهاء
        ▼
      منتهية ✔        (نهائية — لا تُعاد للفتح)

  أي حالة غير نهائية ──► ملغاة ✘   (المدير فقط)
```

**قاعدة الملكية:** التقدّم يملكه **الفني وحده**. المدير يُسنِد ويُلغي، ولا يقود التنفيذ — لأن الحالة ادّعاء عن الواقع الميداني، ومَن ليس في الموقع لا يملك أن يدّعيه.

### 10.2 مسار العمل — تطبيق الفني

```
① الهاتف مقفل ──► إشعار Push: «مهمة جديدة — WO-2026-0042»
                     └─ ضغطة واحدة تفتح المهمة نفسها، لا القائمة

② شاشة المهمة
   ├─ العميل · الفرع · الجهاز (ماركة/موديل/سيريال) · وصف البلاغ · الأولوية
   ├─ [الاتجاهات] ──► خرائط جوجل على إحداثيات الموقع
   ├─ [واتساب المدير] ──► رسالة جاهزة عبر wa.me
   └─ [تاريخ الجهاز] ──► ما حدث لهذه الوحدة سابقًا + حالة الضمان
                          (قراءة فقط، ومقصورة على الأجهزة المُسنَدة إليه)

③ [قبول المهمة]   ──► تُسجَّل بالوقت والإحداثيات
④ [في الطريق]     ──► تُسجَّل بالوقت والإحداثيات ← المدير يُشعَر
⑤ [بدء العمل]     ──► تُسجَّل بالوقت والإحداثيات

⑥ أثناء العمل
   ├─ [عهدتي] ──────► ما في السيارة من قطع، ليقرّر ما يستخدمه
   ├─ [صور] ────────► الكاميرا مباشرة: قبل / بعد
   └─ [التقرير الفني]
        ├─ قراءات مُهيكلة: جهد الدخول · جهد الخروج · نسبة التحميل
        │                  · عدد البطاريات وحالتها · زمن الـ Backup
        ├─ ما تم رصده · ما تم تنفيذه · التوصيات
        └─ قطع الغيار المستخدمة ──► تُخصم من عهدته هو، لا من المخزن الرئيسي

⑦ [توقيع العميل] ──► العميل يوقّع على الشاشة، يُحفَظ مع التقرير باسمه
⑧ [طباعة التقرير] ──► ورقة خدمة تُسلَّم للعميل في الموقع
⑨ [إنهاء المهمة] ──► «منتهية» ← المدير يُشعَر ← تصبح قابلة للفوترة
```

**ثلاثة قرارات تصميمية تخصّ الميدان:**

| القرار | لماذا |
|---|---|
| القراءات **رقمية مُهيكلة** لا نص حر | حتى يمكن لاحقًا رسم اتجاه بطارية بعينها عبر الزيارات — النص الحر يموت في الأرشيف |
| التقرير **يُحفَظ كمسودة أثناء الكتابة** | الفني يكتب في بدروم، والاتصال ينقطع، وإعادة الكتابة تعني ألّا يُكتب التقرير أصلًا |
| **فشل الإشعار لا يُلغي العمل** | العمل يُحفَظ أولًا ثم يُشعَر؛ خادم بريد معطّل يجب ألّا يُفقد تقرير زيارة |

## 11. النظام الخلفي (API) — مسار العمل الداخلي

**المستخدم:** لا أحد مباشرةً — هو ما ينفّذه التطبيقان. وثّقناه لأن **كل قاعدة عمل تعيش هنا**، والواجهتان مجرّد طريقتين لطلبها.

### 11.1 رحلة الطلب الواحد

```
طلب من الواجهة (Bearer Token)
   ▼
① Middleware `auth:sanctum`  ──► رمز صالح؟
   ▼
② Middleware `role:...`      ──► الدور مسموح له بهذا المسار؟ والمستخدم غير موقوف؟
   ▼
③ Controller                 ──► تحقّق من صحة المدخلات (validate) فقط
   │                              لا يكتب حالة ولا يُنشئ قيدًا
   ▼
④ Service (المالك الحصري)     ──► قاعدة العمل + المعاملة (transaction)
   │                              مثال: BillingService يكتب السند وحركة الخزنة معًا
   ▼
⑤ Model events → Observers   ──► الأثر التابع: ترحيل محاسبي · إشعارات · SLA
   ▼
⑥ Resource                   ──► شكل الاستجابة
```

**الفصل الحاسم بين ③ و④:** الـ Controller يتحقق من *الشكل*، والـ Service يتحقق من *المعنى*. «المبلغ رقم موجب» في الـ Controller؛ «المبلغ لا يتجاوز رصيد الخزنة» في الـ Service — لأن الثانية قاعدة عمل تلزم كل مسار يصل إليها، لا مسارًا بعينه.

### 11.2 مسار المال — من المستند إلى الدفتر

```
فاتورة تُصدَر ─┐
سند قبض ───────┤
مصروف ─────────┼──► CashMovement / StockMovement / Invoice
تحويل خزنة ────┤            │
سداد مورّد ────┤            │ (حدث `created` / `updated`)
حركة مخزون ────┘            ▼
                      PostingObserver
                            │
                            ▼
                      LedgerPoster ──► «ماذا يعني هذا المستند بالقيد المزدوج؟»
                            │
                            ▼
                        Ledger ──► قاعدتان لا استثناء لهما:
                            │        · القيد يتوازن أو يُرفَض
                            │        · المستند يُرحَّل مرة واحدة مهما تكرّر الطلب
                            ▼
                    JournalEntry + JournalLine
                            │
                            ▼
                    FinancialReports ──► ميزان · دخل · مركز مالي · أستاذ
```

**لماذا المراقب (Observer) لا الاستدعاء المباشر:** كل مسار يحرّك مالًا ينتهي عند أحد ثلاثة موديلات. مراقبة الموديل تغطّي المسارات الموجودة **والتي ستُضاف لاحقًا**؛ قائمة استدعاءات موزّعة على الخدمات كانت ستُنسى أول مرة يُضاف مسار جديد.

**لماذا فشل الترحيل يُسجَّل ولا يُرمى:** المستندات هي سجلّ الشركة، والدفتر مُشتقّ منها. دليل حسابات ناقص لحظةً يجب ألّا يمنع فوترة زيارة فني. والمقابل الأمين لهذا التنازل أن الفجوة **تُعرَض** — شاشة المحاسبة تقول «كذا مستند لم يصل إلى اليومية»، و`accounting:post` يُلحِقها.

### 11.3 الخدمات ومسؤولياتها الحصرية

| الخدمة | ما لا يجوز لغيرها فعله |
|---|---|
| `TaskWorkflow` | تغيير حالة مهمة |
| `StockLedger` | تحريك كمية أو تعديل متوسط التكلفة |
| `CustodyService` | فتح أو إغلاق عهدة |
| `BillingService` | إصدار فاتورة أو تسجيل تحصيل |
| `SalesService` · `PurchasingService` | نقل عرض/أمر بين حالات دورته |
| `WarrantyService` | حساب صلاحية الغطاء أو البتّ في مطالبة |
| `Ledger` | الكتابة في دفتر اليومية |
| `ChartOfAccounts` | إنشاء حساب نظامي أو ربط خزنة بحساب |

## 15. معايير القبول العامة للمرحلة الحالية

المرحلة تُعتبر **مقبولة** إذا تحقق كل ما يلي:

1. جهاز واحد يمكن تتبّعه من **أمر الشراء ← الاستلام ← البيع ← التركيب ← الضمان ← العقد ← بلاغ ← تقرير فني ← فاتورة ← تحصيل ← قيد محاسبي** دون خروج من النظام.
2. الفني ينهي زيارة كاملة **من موبايله** بما فيها التقرير والصور وتوقيع العميل.
3. **ميزان المراجعة متوازن** في أي لحظة دون تدخل يدوي.
4. لا يوجد فني له عهدة **غير مسجّلة** في أي من الأنواع الثلاثة.
5. لا يستطيع فني الوصول إلى بيانات مهمة أو جهاز أو شاشة مالية ليست له — مُثبَت باختبار `uiguard`.
6. كل مستند مالي له **قيد مقابل** أو سبب موثّق لعدم الترحيل.
